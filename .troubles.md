# Спасение Ковчега — баги, проблемы и недоделки

> Снимок на 2026-06-19. Отмечено: **[крит]** — безопасность/целостность,
> **[баг]** — функциональная ошибка, **[недодел]** — фича не закончена,
> **[ui]** — визуал/UX, **[долг]** — техдолг/инфраструктура.
> «(до сессии)» — существовало раньше; «(новое)» — внесено в этой сессии.

---

## 1. Критические (безопасность и целостность)

### 1.1 [крит] ✅ ИСПРАВЛЕНО — `room_state` раскрывал ответы всем игрокам (до сессии)
Было: `getPublicRoomState()` вырезал только `timerInterval`, в `taskPool` уходили
полные задачи с `correct_answer`/`acceptedAnswers`/`keywords`/`tests`/`pairs`.
**Фикс:** добавлена проекция `publicTask()` — задачи отдаются без полей ответа;
для `interactive_match` колонки остаются, но соответствие перемешивается. Ответы
сверяются только на сервере (`room.taskPool` хранит полные объекты). Подсветка
верного варианта на клиенте берётся из `result` сервера. Проверено смоук-тестом.

### 1.2 [крит] Подмена `userId` → захват чужого профиля (новое)
`userId` приходит с клиента в `join_room` и нигде не верифицируется
(`socket.data.userId`). При этом `member.userId` уходит в `room_state` всем в
комнате. Значит, со-игрок может прочитать чужой `userId` и, переподключившись
с ним, через `update_profile` / `get_my_profile` редактировать и читать **полный
приватный** профиль жертвы (включая скрытые контакты), управлять её друзьями.
**Фикс:** связывать `userId` с секретом (токен профиля в localStorage, не
раскрываемый в `room_state`), либо не слать `userId` в общий стейт, а резолвить
адресацию через серверный маппинг; для мутаций профиля требовать секрет.

### 1.3 [крит] PII и пароли в открытом виде (до сессии + новое)
- Пароли учителей — плейнтекст в памяти (`teachers` Map, `socketHandlers.js`).
- Контакты игроков (почта/VK/GitHub) — плейнтекст в `server/users.json` (новое).
**Фикс:** хешировать пароли; для учебной среды — хотя бы вынести в защищённое
хранилище и не логировать; задокументировать, что `users.json` не для прод-PII.

### 1.4 [крит] ✅ ИСПРАВЛЕНО — импортированные задачи утекали с ответами (новое)
`room.importedTasks` теперь полностью исключён из `room_state`, а задачи пула
проходят через `publicTask()` (см. 1.1).

## 2. Функциональные баги

### 2.1 [баг] ✅ ЧАСТИЧНО — кастомный пул мог сделать победу «по секторам» недостижимой (новое)
Условие конца игры — «все 12 секторов захвачены». Если пул не покрывает все 12
секторов, победа по секторам невозможна (только таймер).
**Сделано:** при фиксации пула сервер считает покрытие и клиент предупреждает
«Покрыто N/12 секторов…». **Осталось (опц.):** пересчитывать win-condition по
фактически достижимым секторам, а не жёстко по 12.

### 2.2 [баг] ✅ ИСПРАВЛЕНО — импорт не валидировал «решаемость» структурных задач (новое)
`sanitizeImportedTask` теперь отбраковывает `full_code` без `entry`/`tests` и
`interactive_match` без `pairs` (≥2) — нерешаемые задачи в каталог не попадают.

### 2.3 [баг] Коллизия ID при импорте пака из другой системы (новое)
В `admin_import_pack` задача с `id`, уже существующим в каталоге, считается
«известной» и просто отмечается — даже если её содержимое другое. Импорт «чужого»
пака с пересекающимися ID выберет не те задачи.
**Фикс:** сверять не только ID, но и контент; либо перенумеровывать
импортируемые задачи и матчить по хешу содержимого.

### 2.4 [баг] Нет лимита на размер импорта (новое)
`admin_import_pack` принимает массив любой длины → можно раздуть
`room.importedTasks` и `room_state`. **Фикс:** ограничить число задач в паке.

## 3. Недоделанные фичи / ограничения

### 3.1 [недодел] Друзья добавляются без согласия (новое)
`add_friend` сразу делает связь взаимной, без запроса/подтверждения. Любой может
односторонне «зафрендить» другого. Нет flow «заявка → принятие».
**Фикс:** опциональные friend-requests по аналогии с join-request.

### 3.2 [недодел] Список друзей всегда публичен (новое)
В `publicView` друзья и `friendCount` видны всем; флага приватности для списка
друзей нет (в отличие от контактов).

### 3.3 [недодел] Профили только у игроков в командах (новое)
Кликнуть имя и открыть мини-профиль можно лишь для тех, кто состоит в команде;
игроки в комнате без команды в `room_state` не отображаются. Учитель профиля не
имеет вовсе.

### 3.4 [недодел] `users.json` и `joinRequests` не очищаются (новое)
- `users.json` растёт безгранично, нет TTL/чистки неактивных профилей.
- `joinRequests` (Map по `roomId`) не удаляется при гибели комнаты (Room Reaper
  чистит комнаты, но не связанные запросы) → мелкая утечка памяти.

### 3.5 [недодел] Имя профиля ≠ позывной (новое)
Отображаемое имя в профиле и игровой позывной независимы и могут различаться —
потенциальная путаница, кого видишь в лобби и в профиле.

### 3.6 [недодел] Мат-фильтр базовый, обходится и даёт ложные срабатывания (новое)
`teamNameFilter.js` — подстрочный матч корней + простой leet. Обходится
homoglyph-атакой (латинские буквы вместо кириллических) и даёт ложные
срабатывания на легитимных словах, содержащих корень. Достаточно для учебной
среды, но не «боевой».

### 3.7 [недодел] Ребрендинг частичный (новое, осознанно)
UI/заголовки переименованы в «Спасение Ковчега», но имя ИИ-антагониста
«Aegis-X» намеренно оставлено в лоре (`App` entry, `GameOver`,
`game_started`) и в `taskDatabase.js` (включая **ответы** задач, напр. ASCII →
«AEGIS»). Полный рескин злодея — отдельная задача (не трогать ответы задач).

## 4. UI / UX

### 4.1 [ui] Светлая тема: подложки модалок остаются тёмными (новое)
Бэкдропы `bg-cyber-dark/90` в светлой теме не инвертируются (намеренно, для
затемнения), но это смотрится контрастно к светлым панелям.

### 4.2 [ui] Низкий контраст неоновых командных цветов на белом (новое)
Цвета команд (например, жёлтый `#ffd60a`, неон) используются как текст/акцент и
на белом фоне читаются плохо. Идентичность важна, но контраст страдает.

### 4.3 [ui] Нет «не спрашивать снова» для подтверждения ответа (новое)
Подтверждение ответа всегда модальное; для длинных серий задач может раздражать.
Возможна опция «подтверждать только тесты (одна попытка)».

### 4.4 [ui] ShipMap: свечение носа/двигателя в светлой теме (новое)
Полупрозрачные голубые круги носа остаются «неоновыми»; не критично.

## 5. Технический долг / инфраструктура

### 5.1 [долг] Нет автоматических тестов
В репозитории нет тест-сьюта. Проверки в этой сессии делались ad-hoc
скриптами через socket.io-client и удалялись. Нужны unit (gameState, фильтры,
sanitize) и интеграционные (socket-flow) тесты.

### 5.2 [долг] Всё состояние в памяти (до сессии, осознанно)
Комнаты, матчи, сессии reconnect, учителя — теряются при рестарте. Профили и
кастомные задачи персистятся в JSON. Для прод нужна БД.

### 5.3 [долг] `types.d.ts` поддерживается вручную
Контракт синхронизируется руками; рассинхрон с кодом возможен. Стоит
генерировать типы или ввести проверку.

### 5.4 [долг] Незакоммиченное изменение `server/customTasks.json`
В рабочем дереве висит правка `customTasks.json` (только удаление завершающего
перевода строки), не относящаяся к фичам. Решить: закоммитить или откатить.

### 5.5 [долг] Дев-серверы запускаются вручную / нет CI
`start.sh` поднимает оба процесса; нет продакшн-пайплайна, healthcheck-мониторинга,
CI-сборки.
